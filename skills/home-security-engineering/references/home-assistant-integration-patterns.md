# Home Assistant Integration Patterns (Alarmo & Prior Art)

Use this when the user is deciding how to architect a Home Assistant integration for a security product — including a future 911-dispatch integration — and needs to know what already exists in the HA ecosystem so it doesn't get rebuilt badly or coupled to the wrong thing.

## The actual integration surface is the `alarm_control_panel` domain, not any one implementation

Home Assistant defines `alarm_control_panel` as an entity domain with a fixed state/service contract (`disarmed`, `armed_home`, `armed_away`, `armed_night`, `armed_vacation`, `pending`, `arming`, `triggered`, plus `alarm_arm_*`/`alarm_disarm`/`alarm_trigger` services). Every implementation — the built-in `manual` platform, Alarmo, or a bridged physical panel — exposes that same contract. **Build a dispatch layer against that domain contract, not against any single implementation's internals.** That's the single most important architectural decision here: it's the difference between an integration that works for every HA user's existing alarm setup and one that only works if they happen to run one specific community project.

## The landscape to know before designing against it

- **Built-in `manual` platform**: HA core's own baseline `alarm_control_panel` — one set of arm/disarm codes, fixed delay times, no per-zone logic. Functional but limited; most serious HA-based alarm setups outgrow it.
- **Alarmo** (HACS): the de facto community standard for a fully-featured, HA-native alarm system — per-zone arm-mode membership (which sensors are active in home/away/night/etc.), per-zone entry/exit delays, per-user codes with individual permissions, an "open sensors" check that blocks arming until relevant zones are clear, and its own event/notification hooks that automations can key off. If the goal is real arm/disarm sophistication without rebuilding a state machine from scratch, Alarmo is the reference implementation to study — and, for many real households, the thing already running that a dispatch integration needs to sit alongside rather than replace.
- **Panel-bridge integrations** (envisalink, AlarmDecoder): bridge an existing *physical, wired* panel (DSC PowerSeries, Honeywell/Ademco Vista) into HA over IP or serial (envisalink's Envisalink module; AlarmDecoder's AD2USB/AD2PI hardware), exposing the panel's own zones and state as HA entities. Relevant for any household with a legacy wired system — a dispatch integration should be able to treat a bridged panel's `alarm_control_panel` entity the same way it treats Alarmo's, since it's the same domain contract.
- **Konnected**: a different layer entirely — a Wi-Fi board that wires directly into a legacy panel's zone terminals (or standalone sensors) and exposes each zone as an HA `binary_sensor`/`switch`, without providing arm-state logic itself. Typically paired with Alarmo or the built-in platform for the actual state machine — worth knowing as the "get the sensors in" half of a legacy-panel retrofit, distinct from the "manage arm state" half.
- **Cloud-mirror integrations** (SimpliSafe, Abode, Ring, Total Connect–style bridges): expose a commercial cloud-monitored system's state *into* HA, but the actual arm/disarm authority and logic live in the vendor's cloud, not in HA. Structurally different trust model from Alarmo/manual/bridged-panel — ties directly to the cloud-dependency-as-single-point-of-failure risk in `threat-modeling.md`. Don't design a dispatch trigger path as if these behave like a local-authority system; a cloud outage on the vendor's side can mean HA's mirrored state is stale exactly when it matters most.

## Where a dispatch integration should actually plug in

- **Subscribe to `alarm_control_panel` state changes, filtered to `triggered`**, as the hand-off point into the verification ladder in `dispatch-integration.md` — regardless of which backend produced it. A `triggered` state already means *that system's own* entry-delay/no-valid-disarm logic ran (real signal, not raw), but it's still one local system's automated determination — not equivalent to human, video, or cross-system corroboration. Treat it as the floor of the verification ladder, not the ceiling, before anything approaching an actual dispatch decision.
- **Prefer a companion integration over a fork.** Forking Alarmo (or any panel-bridge integration) to bolt on dispatch logic fragments the ecosystem, loses upstream maintenance, and turns every future upstream update into a manual merge. A companion integration that listens on the event bus and, where useful, ships an official Alarmo-compatible automation blueprint is the pattern that fits how the HA community already extends systems.
- **Pull richer context opportunistically, not as a hard dependency.** Where the backend happens to be Alarmo specifically, its per-zone open-sensor data and per-user code identity are worth reading for extra dispatch context (which zone, who disarmed or didn't) — but design the core trigger path so it degrades gracefully to "generic `triggered` state, no extra context" for a bridged panel or the `manual` platform.
- **Respect licensing/attribution** on anything referenced or adapted from Alarmo or the other named projects — they're independently maintained open-source projects with their own licenses; check the current license and any attribution requirements before bundling, redistributing, or forking their code rather than assuming permissive terms apply.
- **Distribution**: this ecosystem (Alarmo, envisalink, AlarmDecoder, Konnected) is predominantly installed via **HACS** (Home Assistant Community Store), not HA core's built-in integrations list — plan a new integration's own distribution and update path around HACS conventions if the goal is to reach the same install base.

## Out of scope

This reference is about architecting against and interoperating with existing HA alarm integrations. It's not a guide to reverse-engineering a *specific* commercial panel's proprietary protocol without authorization, or to bypassing a cloud-mirror integration's vendor authentication — see `access-control-and-locks.md` and `threat-modeling.md` for the general boundary on defeating a named real system.
